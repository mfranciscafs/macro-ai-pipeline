1) Migrations — entendeste quase tudo certo

Sim: a lógica é mesmo essa.

Ideia principal

Tens ficheiros versionados no Git que descrevem mudanças ao schema da base de dados.

Exemplo:

migrations/
├── 001_create_macro_data_table.sql
├── 002_add_source_column.sql
└── 003_create_index_on_year.sql

Cada ficheiro diz à BD o que fazer.

Fluxo mental correto
Hoje

Queres criar a tabela:

001_create_macro_data_table.sql
Amanhã

Queres adicionar uma coluna:

002_add_unit_column.sql
Depois

Queres criar um índice:

003_add_index.sql
Porque isto é poderoso

Porque depois alguém faz:

git clone ...

e corre migrations, e a base de dados fica no mesmo estado esperado.

Ou seja, sim:

fica muito mais reproduzível, auditável e controlado

Mas atenção

As migrations são para schema changes, não para “inserir dados normais do pipeline”.

Ou seja:

migrations servem para:

criar tabela

alterar coluna

criar índice

mudar constraints

migrations NÃO servem para:

inserir GDP de Portugal de 2024

carregar dados diários do pipeline
